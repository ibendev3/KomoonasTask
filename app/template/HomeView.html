Development Document

Overview

Handling with the assignment and it's details was pretty challenging, giving the amount of time, the unfamiliar area of web dynamic charts and the large amount of data coming from distanced resource needed to be analyzed at some point.
With that said, I had to do some pre-coding research and design. the decisions I took at the beginning has changed with the development but they made the assignment become more clear.

Architecture

The application is running under NodeJS web server, and architect to be a single-page application using the capabilities of the front end solution - AngularJS - Taking care of client-side routing, data binding and dynamic events occurring in the page. The visualization charts are being handled by an open source library by Google (Google Charts) which I chose because after testing a few other options(D3, Charts.js) Google's was the easiest to customise to a specific need and has the best support with Angular, but still - I'm not 100% sure it's the best solution for a larger application. Look and feel is with components and styling of Twitter-Bootstrap Which I find amazing when you need a quick layout and objects on screen.

Data flow

The data flow performed by the following:

  1. Webserver is starting
  2. Webserver send a request to the DataSource holding the JSON file
  3. Data is parsed, analyzed, set to relevant data structures and cached on the server side
  4. When needed on the page - client call the data from the server (where it cached and minimized)
  5. Data is injected to the client objects and charts

Visualization

The application present 2 pages of different visualization:
The first one is the daily volume - It display a linear chart of a chosen date and the data displayed compare between inited, served and default on hour during the day. Tag ids can be filtered or display all.
The second one is displaying data regarding strategies used by Komoona algorithms, for a chosen tag it gives a view of all strategies attempted and what happened with the requests. here you can filter by amount of requests inited (gives the ability to observe on strategies with low usage).

Pitfall

The most challenging pitfall with the assignment was dealing with the data, coming from a distanced source, all at once and need to be structured in a way I could "play" with easily once an event is triggered on the page. I think that if I had a control over the data source I would probably load it in chunks of days, though that if the historical data is static, it only needs to be loaded once, when the server launches, and then, just adding the offset data every single hour.
The second pitfall was choosing and building the environment, at first I wanted to save time so I chose to work on a client side application only, But I wasn't satisfied with the performance of the data analytic, so I had to add a server in order to process the data only once, not per request.
